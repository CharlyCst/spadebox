# Maintain Documentation

This skill explains how to keep all public-facing documentation in sync after changes to the Spadebox codebase.

## Documentation Architecture

Understanding where documentation lives is essential before making any change.

### Source of truth: `crates/spadebox-core/src/tools/`

Each tool is defined in its own file (`read.rs`, `write.rs`, `edit.rs`, `glob.rs`, `grep.rs`). Every tool file contains:

- A `Params` struct with **field-level doc comments** — these flow into the tool's JSON Schema and are what agents and
  users see as parameter descriptions.
- A `Tool` impl with two constants:
  - `NAME`: the canonical tool name (e.g. `"read_file"`)
  - `DESCRIPTION`: the human-readable description exposed to all interfaces

These two constants and the `Params` doc comments are the **single source of truth** for tool documentation. The MCP
server and JSON schema both derive from them automatically — never duplicate this information elsewhere.

### Derived / auto-generated

- **`js/index.d.ts`** — auto-generated by NAPI-RS on `just build`. Never edit it by hand. Doc comments from
  `js/src/lib.rs` flow into it automatically.
- **JSON Schema** (`inputSchema` in `SbTool`) — generated at runtime via the `#[derive(JsonSchema)]` on each `Params`
  struct.
- **MCP tool list** — the MCP server calls `enabled_tools(sandbox)` from `spadebox-core` and converts each `ToolDef` to
  an MCP tool definition; no manual sync needed.

### Manually maintained

- **`js/src/lib.rs`** — NAPI-RS bindings. Contains hand-written doc comments for the `SpadeBox` class and each
  convenience method (`read_file`, `write_file`, etc.). These are **not** auto-derived from core; they must be updated
  manually. Use **camelCase** for all parameter and field references in these doc comments because NAPI-RS converts
  snake_case identifiers to camelCase in the generated bindings. See the comment at the top of the file for details.
- **`python/src/lib.rs`** — PyO3 bindings. Contains hand-written doc comments for the `SpadeBox` class and each
  convenience method. These are surfaced directly as Python `help()` text and are **not** auto-derived from core; they
  must be updated manually. Use **snake_case** for all parameter and field references, matching Python conventions.
- **`rust/src/lib.rs`** — Public Rust bindings. Contains hand-written doc comments for the `SpadeBox` struct and each
  convenience method. These flow into `cargo doc` output and are **not** auto-derived from core; they must be updated
  manually. Use **snake_case** for all parameter and field references, matching Rust conventions.
- **`README.md`** — high-level overview listing available tools and usage examples for each interface. Shared across the
  repo root, `js/`, and `python/` via a symlink.

## Checklist by Change Type

### Adding or renaming a parameter on an existing tool

1. **`crates/spadebox-core/src/tools/<tool>.rs`**
   - Add or rename the field on the `Params` struct.
   - Write a clear `///` doc comment on the new field.
   - Update `const DESCRIPTION` if the tool's overall behaviour description needs to mention the new parameter.

2. **`js/src/lib.rs`**
   - Update the corresponding convenience method signature (add the new parameter as `Option<T>` if it is optional).
   - Update the method's doc comment to mention the new parameter, using camelCase for its name.
   - Pass the new parameter through to the `Params` struct inside the method body.

3. **`python/src/lib.rs`**
   - Update the corresponding convenience method signature (add the new parameter as `Option<T>` if it is optional, with
     a `#[pyo3(signature)]` attribute to set the default).
   - Update the method's doc comment to mention the new parameter, using snake_case for its name.
   - Pass the new parameter through to the `Params` struct inside the method body.

4. **`rust/src/lib.rs`**
   - Update the corresponding convenience method signature (add the new parameter as `Option<T>` if it is optional).
   - Update the method's doc comment to mention the new parameter, using snake_case for its name.
   - Pass the new parameter through to the `Params` struct inside the method body.

5. **No changes needed** in the MCP server, `js/index.d.ts`, or JSON Schema — they all update automatically on the next
   build.

### Changing a tool description

1. **`crates/spadebox-core/src/tools/<tool>.rs`** — update `const DESCRIPTION`. The MCP server picks this up
   automatically.

2. **`js/src/lib.rs`** — update the doc comment on the corresponding convenience method to stay consistent. Remember
   camelCase.

3. **`python/src/lib.rs`** — update the doc comment on the corresponding convenience method to stay consistent. Remember
   snake_case.

4. **`rust/src/lib.rs`** — update the doc comment on the corresponding convenience method to stay consistent. Remember
   snake_case.

5. **Regenerate the doc website tool pages** — from `website/` run `deno task generate`.

### Adding a new tool

1. **Create `crates/spadebox-core/src/tools/<name>.rs`** following the existing pattern:
   - Define a `Params` struct with `#[derive(Debug, Deserialize, JsonSchema)]` and field-level doc comments.
   - Implement the `Tool` trait with `NAME`, `DESCRIPTION`, and `run`.

2. **`crates/spadebox-core/src/tools/mod.rs`**
   - `pub mod <name>;` and re-export the new `*Tool` and `*Params` types.
   - Add a `NewTool::def()` call inside `enabled_tools()`, guarded by the appropriate `sandbox.<subsystem>.is_enabled()`
     check.
   - Add a `NewTool::NAME if sandbox.<subsystem>.is_enabled() => NewTool::call_json(sandbox, params_json).await` arm to
     the `call_tool` match.

3. **`crates/spadebox-mcp/src/main.rs`** — **no changes needed**. The MCP server calls `enabled_tools(sandbox)` which
   automatically picks up any tool registered in `enabled_tools()` in `tools/mod.rs`, and delegates all dispatch to
   `spadebox_core::call_tool`.

4. **`js/src/lib.rs`**
   - Add a convenience method (`#[napi] pub async fn <name>(...)`) with a clear doc comment using camelCase. The
     `tools()` method calls `enabled_tools(&self.inner)` automatically — no extra registration needed.

5. **`python/src/lib.rs`**
   - Add a convenience method (`pub fn <name>(...)`) annotated with `#[pyo3(signature = (...))]` for optional
     parameters, with a clear doc comment using snake_case. The `tools()` method calls `enabled_tools(&self.inner)`
     automatically — no extra registration needed.
   - Wrap async execution with `py.detach(|| { runtime.block_on(async { ... }) })`.

6. **`rust/src/lib.rs`**
   - Add a convenience method (`pub async fn <name>(...)`) with a clear doc comment using snake_case. The `tools()`
     method calls `enabled_tools(&self.inner)` automatically — no extra registration needed.

7. **`README.md`** — add the new tool name to the tool list.

7. **Regenerate the doc website tool pages** — from the `website/` directory run:
   ```
   deno task generate
   ```
   This calls the CLI for every tool and rewrites `website/docs/tools/<name>.md`. The new tool's page is created
   automatically; no manual file editing needed.

### Adding a new language binding (e.g. Python)

The goal is to mirror the pattern established by `js/`:

1. Create a new directory (e.g. `python/`) with its own build setup.
2. Use the core crate (`spadebox-core`) as the implementation backend.
3. For each convenience wrapper function, write doc comments that:
   - State what the function does and name the underlying tool it calls.
   - Use the target language's naming conventions for parameter references (e.g. `snake_case` for Python, `camelCase`
     for JavaScript).
4. Do **not** copy `NAME`, `DESCRIPTION`, or parameter descriptions from the core — let the binding derive or call them
   at runtime/generation time, just as the MCP server and JS bindings do.
5. Register the tool in `enabled_tools()` in `crates/spadebox-core/src/tools/mod.rs` so `tools()` (which calls
   `enabled_tools`) returns all tools automatically.
6. Update `README.md` to mention the new binding with a usage example.
