# Maintain Documentation

This skill explains how to keep all public-facing documentation in sync after
changes to the Spadebox codebase.

## Documentation Architecture

Understanding where documentation lives is essential before making any change.

### Source of truth: `crates/spadebox-core/src/tools/`

Each tool is defined in its own file (`read.rs`, `write.rs`, `edit.rs`,
`glob.rs`, `grep.rs`). Every tool file contains:

- A `Params` struct with **field-level doc comments** — these flow into the
  tool's JSON Schema and are what agents and users see as parameter descriptions.
- A `Tool` impl with two constants:
  - `NAME`: the canonical tool name (e.g. `"read_file"`)
  - `DESCRIPTION`: the human-readable description exposed to all interfaces

These two constants and the `Params` doc comments are the **single source of
truth** for tool documentation. The MCP server and JSON schema both derive from
them automatically — never duplicate this information elsewhere.

### Derived / auto-generated

- **`js/index.d.ts`** — auto-generated by NAPI-RS on `just build`. Never edit
  it by hand. Doc comments from `js/src/lib.rs` flow into it automatically.
- **JSON Schema** (`inputSchema` in `SbTool`) — generated at runtime via the
  `#[derive(JsonSchema)]` on each `Params` struct.
- **MCP tool list** — the MCP server calls `T::NAME`, `T::DESCRIPTION`, and
  `T::schema()` directly; no manual sync needed.

### Manually maintained

- **`js/src/lib.rs`** — NAPI-RS bindings. Contains hand-written doc comments for
  the `SpadeBox` class and each convenience method (`read_file`, `write_file`,
  etc.). These are **not** auto-derived from core; they must be updated manually.
  Use **camelCase** for all parameter and field references in these doc comments
  because NAPI-RS converts snake_case identifiers to camelCase in the generated
  bindings. See the comment at the top of the file for details.
- **`README.md`** — high-level overview listing available tools. Shared across
  the repo root and `js/` via a symlink.

## Checklist by Change Type

### Adding or renaming a parameter on an existing tool

1. **`crates/spadebox-core/src/tools/<tool>.rs`**
   - Add or rename the field on the `Params` struct.
   - Write a clear `///` doc comment on the new field.
   - Update `const DESCRIPTION` if the tool's overall behaviour description
     needs to mention the new parameter.

2. **`js/src/lib.rs`**
   - Update the corresponding convenience method signature (add the new
     parameter as `Option<T>` if it is optional).
   - Update the method's doc comment to mention the new parameter, using
     camelCase for its name.
   - Pass the new parameter through to the `Params` struct inside the method
     body.

3. **No changes needed** in the MCP server, `js/index.d.ts`, or JSON Schema —
   they all update automatically on the next build.

### Changing a tool description

1. **`crates/spadebox-core/src/tools/<tool>.rs`** — update `const DESCRIPTION`.
   The MCP server picks this up automatically.

2. **`js/src/lib.rs`** — update the doc comment on the corresponding
   convenience method to stay consistent. Remember camelCase.

### Adding a new tool

1. **Create `crates/spadebox-core/src/tools/<name>.rs`** following the
   existing pattern:
   - Define a `Params` struct with `#[derive(Debug, Deserialize, JsonSchema)]`
     and field-level doc comments.
   - Implement the `Tool` trait with `NAME`, `DESCRIPTION`, and `run`.

2. **`crates/spadebox-core/src/tools/mod.rs`**
   - `pub mod <name>;` and re-export the new `*Tool` and `*Params` types.
   - Add a `NewTool::NAME => NewTool::call_json(sandbox, params_json).await`
     arm to the `call_tool` match.

3. **`crates/spadebox-mcp/src/main.rs`**
   - Import the new `*Tool` type.
   - Add `mcp_tool::<NewTool>()` to the `list_tools` vec.
   - Add a dispatch arm in `call_tool` (if the MCP server has its own match,
     otherwise it delegates to `spadebox_core::call_tool` and needs no change).

4. **`js/src/lib.rs`**
   - Add `make_tool_entry::<NewTool>()` to the `build_tools` array.
   - Add a convenience method (`#[napi] pub async fn <name>(...)`) with a
     clear doc comment using camelCase.

5. **`README.md`** — add the new tool name to the tool list.

### Adding a new language binding (e.g. Python)

The goal is to mirror the pattern established by `js/`:

1. Create a new directory (e.g. `python/`) with its own build setup.
2. Use the core crate (`spadebox-core`) as the implementation backend.
3. For each convenience wrapper function, write doc comments that:
   - State what the function does and name the underlying tool it calls.
   - Use the target language's naming conventions for parameter references
     (e.g. `snake_case` for Python, `camelCase` for JavaScript).
4. Do **not** copy `NAME`, `DESCRIPTION`, or parameter descriptions from the
   core — let the binding derive or call them at runtime/generation time, just
   as the MCP server and JS bindings do.
5. Add a `make_tool_entry`-equivalent registration so `tools()` returns all
   five tools.
6. Update `README.md` to mention the new binding.

## Keeping This Skill Up to Date

If, while following this skill, you notice that any part of it no longer
matches the actual codebase (e.g. a file has moved, a new interface has been
added, or the registration pattern has changed), flag it to the user before
finishing. Briefly describe what is outdated and suggest the corrected text, so
the skill can be updated to stay accurate.
